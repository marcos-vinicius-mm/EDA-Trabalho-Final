\section{Introdução}

Este trabalho foi desenvolvido como projeto final da disciplina \textbf{QXD0115 -- Estrutura
de Dados Avançada} e tem como objetivo a implementação de um programa que realiza a
contagem de frequência de palavras em arquivos de texto, utilizando quatro diferentes
estruturas de dados do tipo dicionário:

\begin{itemize}
    \item Árvore AVL completamente iterativa;
    \item Árvore Rubro-Negra completamente iterativa;
    \item Tabela Hash com encadeamento exterior (\textit{separate chaining});
    \item Tabela Hash com endereçamento aberto (\textit{open addressing}) por sondagem linear.
\end{itemize}

Além de gerar o CSV com as palavras e suas frequências em ordem alfabética, o programa
coleta métricas de desempenho de cada estrutura -- número de comparações de chave,
rotações, colisões e tempo de construção -- permitindo uma análise comparativa objetiva
entre as abordagens.

\subsection{Motivação}

A contagem de frequência de palavras é um problema clássico em ciência da computação,
com aplicações em análise de texto, recuperação de informação, processamento de linguagem
natural e compressão de dados. O uso de dicionários é natural para esse problema pois
permitem inserção e busca eficiente por chave. Diferentes implementações de dicionário
apresentam comportamentos distintos de desempenho dependendo do volume de dados e da
distribuição das chaves, o que torna esse projeto um cenário ideal para comparação
empírica das estruturas estudadas em aula.

\subsection{Objetivos}

\begin{itemize}
    \item Implementar quatro dicionários genéricos com templates C++ herdando de uma
          interface abstrata comum (\texttt{Dictionary<Key,Value>});
    \item Implementar AVL e Rubro-Negra \textbf{sem recursão}, usando apenas laços iterativos;
    \item Processar arquivos \texttt{.txt} realizando tokenização, normalização e contagem;
    \item Coletar e comparar métricas de desempenho das quatro estruturas;
    \item Disponibilizar uma interface de linha de comando clara e documentada.
\end{itemize}

% 2. FUNDAMENTAÇÃO TEÓRICA
% ============================================================
\section{Fundamentação Teórica}

\subsection{Dicionários}

Um dicionário é uma estrutura de dados que armazena pares \texttt{(chave, valor)}, onde
cada chave é única e permite acesso direto ao valor associado. As operações fundamentais
são inserção, busca, atualização e remoção, todas com complexidade média $O(\log n)$ em
árvores balanceadas ou $O(1)$ em tabelas hash.

\subsection{Árvore AVL}

A Árvore AVL, proposta por Adelson-Velsky e Landis em 1962, é uma árvore binária de
busca que mantém o \textbf{fator de balanceamento} de cada nó entre $-1$ e $+1$, onde:

\begin{equation}
    bf(n) = h(\text{subárvore esquerda}) - h(\text{subárvore direita})
\end{equation}

Quando uma inserção ou remoção viola essa propriedade, rotações simples (LL, RR) ou
duplas (LR, RL) são realizadas para restaurar o balanceamento. Todas as operações têm
complexidade $O(\log n)$ no pior caso, com $h \leq 1{,}44 \log_2(n+2)$.

A implementação deste trabalho é \textbf{completamente iterativa}: o ponteiro
\texttt{parent} em cada nó permite subir pelo caminho de inserção sem pilha de chamadas,
e o método \texttt{balance\_upwards()} realiza o rebalanceamento em um laço \texttt{while}.

\subsection{Árvore Rubro-Negra}

A Árvore Rubro-Negra é uma árvore binária de busca balanceada que mantém cinco
propriedades cromáticas, garantindo que o caminho mais longo até uma folha seja no
máximo o dobro do mais curto. Isso implica altura máxima $2\log_2(n+1)$.

O algoritmo de inserção (\texttt{insert\_fixup}) trata três casos simétricos baseados
na cor do tio do nó inserido, e o de remoção (\texttt{delete\_fixup}) trata quatro
casos simétricos. Ambos foram implementados de forma iterativa seguindo o algoritmo
de Cormen et al. (CLRS).

A estrutura usa um \textbf{nó sentinela NIL} compartilhado por todos os ponteiros nulos,
eliminando verificações de \texttt{nullptr} e simplificando o código.

\subsection{Tabela Hash com Encadeamento Exterior}

Cada posição da tabela é uma lista encadeada de pares \texttt{(chave, valor)}. Uma
colisão é resolvida inserindo o novo elemento no final da lista do bucket correspondente.
O fator de carga $\alpha = n/m$ (onde $n$ é o número de elementos e $m$ o número de
buckets) controla a eficiência: quando $\alpha$ ultrapassa o limite configurado, um
\textit{rehash} dobra o tamanho da tabela.

A função de hash utiliza o método da divisão: $h(k) = \texttt{hash}(k) \bmod m$, onde
$m$ é sempre um número primo para melhor distribuição.

\subsection{Tabela Hash com Endereçamento Aberto}

Todos os elementos são armazenados na própria tabela. Colisões são resolvidas por
\textbf{sondagem linear}: $h(k, i) = (h'(k) + i) \bmod m$. Remoções usam
\textit{lazy deletion} -- o slot é marcado como \texttt{DELETED} em vez de ser
esvaziado, preservando as sequências de sondagem para buscas futuras.

O fator de carga máximo é limitado a $0{,}5$ para evitar degradação de desempenho,
e o rehash reconstrói a tabela reinserindo todos os elementos ativos.

\subsection{Interface Dictionary}

Todas as estruturas herdam da classe abstrata \texttt{Dictionary<Key, Value>}, que
define o contrato comum:

\begin{lstlisting}[caption={Interface Dictionary (simplificada).}]
template <typename Key, typename Value>
class Dictionary {
public:
    virtual bool   insert(const Key& k, const Value& v) = 0;
    virtual Value& at(const Key& k)                     = 0;
    virtual void   update(const Key& k, const Value& v) = 0;
    virtual bool   remove(const Key& k)                 = 0;
    virtual bool   contains(const Key& k) const         = 0;
    virtual size_t size()  const                        = 0;
    virtual bool   empty() const                        = 0;
    virtual void   clear()                              = 0;
    virtual std::vector<std::pair<Key,Value>>
                   to_sorted_vector() const             = 0;
    virtual void   print_csv(std::ostream& os) const    = 0;
    virtual const Metrics& metrics() const              = 0;
    virtual void   reset_metrics()                      = 0;
};
\end{lstlisting}

A estrutura \texttt{Metrics} acumula todas as métricas de desempenho:

\begin{lstlisting}[caption={Estrutura Metrics.}]
struct Metrics {
    long long comparisons  = 0; // Comparacoes de chave
    long long rotations    = 0; // Rotacoes (AVL / Rubro-Negra)
    long long collisions   = 0; // Colisoes (Hash)
    long long reinsertions = 0; // Re-insercoes no rehash
    double    elapsed_ms   = 0.0;
};
\end{lstlisting}

\subsection{Processamento do Texto}

A tokenização do arquivo segue as seguintes regras, implementadas em \texttt{tokenize()}:

\begin{enumerate}
    \item Espaços e pontuação são tratados como separadores de palavras;
    \item Todas as letras maiúsculas ASCII são convertidas para minúsculas;
    \item O hífen \textbf{no interior} de uma palavra é preservado
          (ex.: \textit{mostrá-lo} permanece como uma entrada única);
    \item O hífen \textbf{isolado} ou no início/fim de token é descartado
          (ex.: travessão de diálogo);
    \item Caracteres acentuados (valor $> 127$) são mantidos intactos.
\end{enumerate}

\subsection{Construção da Tabela de Frequências}

O método \texttt{build\_frequency()} é genérico e funciona com qualquer implementação
de \texttt{Dictionary}:

\begin{lstlisting}[caption={Construção da tabela de frequências.}]
static void build_frequency(Dictionary<std::string, int>& dict,
                             const std::vector<std::string>& words) {
    for (const auto& w : words) {
        if (!dict.insert(w, 1)) { // false = chave ja existia
            dict.at(w)++;
        }
    }
}
\end{lstlisting}

\subsection{Análise dos Resultados}

\subsubsection{Árvores AVL vs. Rubro-Negra}

Ambas as árvores garantem $O(\log n)$ por operação. A AVL mantém um balanceamento mais
rígido (diferença de altura máxima 1), o que resulta em menor número de comparações na
busca, mas potencialmente mais rotações durante a inserção. A Rubro-Negra admite
desbalanceamento ligeiramente maior mas realiza menos rotações em média, favorecendo
cargas de trabalho com muitas inserções.

\subsubsection{Hash Encadeamento vs. Endereçamento Aberto}

O encadeamento exterior tolera fatores de carga mais altos sem degradação severa, pois
colisões geram listas locais. O endereçamento aberto é mais sensível ao fator de carga
-- por isso o limite foi configurado em $0{,}5$ -- mas tem melhor localidade de cache
pois todos os dados ficam em um único vetor contíguo.

\subsubsection{Árvores vs. Hash}

As tabelas hash apresentam menor número de comparações de chave na construção, pois o
acesso ao bucket é $O(1)$ e as comparações ocorrem apenas dentro do bucket. As árvores,
por outro lado, realizam $O(\log n)$ comparações por operação mas garantem saída em
ordem alfabética sem custo adicional de ordenação, o que é vantajoso para a geração
do CSV.

\section{Conclusão}

Este trabalho permitiu a implementação prática e a análise comparativa de quatro
estruturas de dados do tipo dicionário, aplicadas ao problema real de contagem de
frequência de palavras em textos literários.

As principais contribuições e aprendizados foram:

\begin{itemize}
    \item A implementação iterativa da AVL e da Rubro-Negra mostrou que é plenamente
          possível evitar recursão utilizando ponteiros para o pai e algoritmos de
          subida iterativa pelo caminho de inserção/remoção;

    \item O uso de uma interface abstrata \texttt{Dictionary<Key, Value>} com
          \textit{templates} C++ permitiu que o código de processamento do texto
          fosse completamente independente da estrutura de dados subjacente;

    \item As tabelas hash apresentaram menor tempo de construção absoluto, mas as
          árvores eliminaram a necessidade de ordenação para gerar o CSV, compensando
          parte dessa diferença;

    \item A Rubro-Negra realizou menos rotações que a AVL, confirmando a teoria de
          que seu critério de balanceamento menos rígido favorece cargas com muitas
          inserções.
\end{itemize}

Como dificuldades, destacam-se a implementação iterativa do \texttt{delete\_fixup} da
Rubro-Negra (os quatro casos simétricos com \texttt{transplant} são delicados de
depurar sem recursão) e o tratamento correto do nó sentinela NIL, cujo ponteiro
\texttt{parent} precisa ser atualizado mesmo quando $x$ é o próprio NIL.
